バグや問題の根本原因を体系的に調査します。修正前に原因特定が必須。

$ARGUMENTS

## ルール

- **根本原因調査なしに修正コードを書くことは禁止**
- 3回同じアプローチで失敗したらSTOPしてユーザーに報告
- 推測で修正しない。証拠に基づいて修正する

---

## Phase 1: 症状の記録

1. エラーメッセージ、スタックトレース、再現手順を記録する
2. 「いつから起きているか」を git log で特定する：
   ```bash
   git log --oneline -20 -- <affected-files>
   ```
3. 再現できるか確認する。再現できない場合はより多くの証拠を集める

**出力：** 症状の要約と、直近で何が変わったかの記録

---

## Phase 2: 仮説立案

1. 最大3つの仮説を立てる
2. 各仮説の検証方法を明記する

```
仮説1: [具体的な原因の推測]
  検証方法: [どうやって確認するか]
仮説2: [具体的な原因の推測]
  検証方法: [どうやって確認するか]
仮説3: [具体的な原因の推測]
  検証方法: [どうやって確認するか]
```

---

## Phase 3: 仮説検証

1. 各仮説を1つずつ検証する（コードを読む、ログを確認、最小再現を作る）
2. 検証結果を記録する
3. **3仮説すべて外れた場合 → STOP**。ユーザーにエスカレーションする：
   ```
   3つの仮説を検証しましたが、いずれも原因ではありませんでした。

   A) 新しい仮説で調査を続行: [仮説の概要]
   B) 人間のレビューにエスカレーション
   C) ログを追加して次回の発生時に捕捉
   ```

**レッドフラグ（これが出たら立ち止まる）：**
- 「とりあえずの修正」→ 正しく直すかエスカレーション
- データフローを追跡する前に修正を提案 → 推測している
- 修正するたびに別の問題が出る → コードではなくアーキテクチャの問題

---

## Phase 4: 修正実装

原因特定後のみ修正に着手する。

1. **根本原因を修正する。** 症状ではなく原因を直す。最小限の変更にする
2. 修正は1コミット1修正にする（git revert 可能にするため）
3. 修正後にテスト実行で検証する
4. **5ファイル以上に変更が及ぶ場合** → ユーザーに確認してから進める

---

## Phase 5: 報告

修正完了後、以下の構造で報告する：

```
調査報告
════════════════════════════════════════
根本原因:    [何が実際に問題だったか]
修正内容:    [何を変更したか、ファイル:行の参照付き]
影響範囲:    [この修正が影響する範囲]
再発防止策:  [あれば記載]
ステータス:  DONE | DONE_WITH_CONCERNS | BLOCKED
════════════════════════════════════════
```

- **DONE** — 原因特定、修正済み、テスト通過
- **DONE_WITH_CONCERNS** — 修正したが完全な検証ができない（間欠的なバグ等）
- **BLOCKED** — 調査後も原因不明、エスカレーション済み
