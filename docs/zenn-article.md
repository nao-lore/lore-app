## この記事の対象読者

- Claude Codeを使っている、または導入を検討している開発者
- AIコーディングツールで実用レベルのアプリを作りたい人
- 「バイブコーディング」の先にある、品質を担保する手法に興味がある人

筆者はプログラミング未経験（21歳、大学中退）で、Claude Codeを使ってPWAを開発しました。この記事では「未経験でもできた」という話ではなく、**非エンジニアがプロダクション品質を達成するために使った具体的な手法**を技術的に掘り下げます。

## 1. アーキテクチャ設計をAIに任せる際の勘所

Claude Codeに「チャット履歴を構造化するアプリを作りたい」と伝えた結果、以下の構成が提案されました。

```
src/
├── components/    # 18の抽出コンポーネント
├── hooks/         # 8カスタムフック
├── storage/       # 6ドメインモジュール（core, logs, todos, projects, masterNotes, settings）
├── utils/         # 7ユーティリティモジュール
├── transform.ts   # AI処理パイプライン
├── chunkEngine.ts # 長文会話の分割エンジン
├── prompts.ts     # 全プロンプト集約
├── schemas.ts     # Zodスキーマ定義
└── i18n.ts        # 国際化
```

ここで重要だったのは、**提案をそのまま受け入れなかったこと**です。最初のアーキテクチャには以下の問題がありました:

- ストレージが単一モジュールに集約されていて、関心の分離ができていなかった
- プロンプトがtransform.tsに直書きされていた
- バリデーションがなかった

これらを「なぜこの構造なのか」と質問し、理由を理解した上で「ストレージをドメインごとに分割してほしい」「プロンプトは別ファイルに集約してほしい」「AI応答にはZodスキーマでバリデーションをかけてほしい」と指示し直しました。

**ポイント:** Claude Codeの初手のアーキテクチャは70〜80点。残りの20〜30点は「なぜ？」を繰り返すレビューで埋まる。

## 2. チャンキングエンジン — 長文会話の処理

Loreの核心機能は、数千メッセージの会話を受け取り、意思決定・TODO・ブロッカー・進捗を構造化データに変換する部分です。

課題は**トークン制限**。長い会話はそのままAPIに渡せません。

Claude Codeと試行錯誤して作ったアプローチ:

1. 会話をセマンティックなチャンクに分割（話題の転換点で切る）
2. 各チャンクに対して個別に構造化抽出を実行
3. 抽出結果をマージ・重複排除
4. 最終的にZodスキーマでバリデーション

```typescript
// chunkEngine.ts の概念的な流れ
const chunks = splitBySemanticBoundary(conversation);
const results = await Promise.all(
  chunks.map(chunk => extractStructuredData(chunk))
);
const merged = deduplicateAndMerge(results);
const validated = schema.parse(merged);
```

ファジー重複排除（`fuzzyDedup`ユーティリティ）は、異なるチャンクから同じTODOが抽出されるケースへの対策です。完全一致ではなく類似度で判定します。

**学び:** AIに「長い文章を処理する仕組みを作って」と曖昧に言うより、「トークン制限があるから分割が必要」「分割すると重複が出るから重複排除が必要」と制約を明示する方がよい設計になる。

## 3. テスト戦略 — 非エンジニアにとってのセーフティネット

780本のテストは、テスト駆動開発の信念から書いたわけではありません。**自分がコードを完全に理解していないから**書きました。

### テストの書き方

Claude Codeに以下のルールを設定しました（CLAUDE.mdに記載）:

```
- Tests: vitest, aim for 500+. Mock localStorage in storage tests
```

新機能を追加するたびに「この機能のテストも書いて」と指示。Claude Codeは通常、ハッピーパスだけ書くので、「エッジケース（空文字列、null、巨大入力）もテストして」と追加で指示します。

### テストが救った実例

18個のClaude Codeエージェントを並列で走らせたとき、Agent Aが修正したコードをAgent Bが別の修正で壊すケースが頻発しました。マージ後に`npx vitest run`を回すと、壊れた箇所がすぐにわかる。テストがなかったら、18エージェントの並列運用は不可能だったと思います。

### CLAUDE.mdによるVerification Gate

```markdown
## Verification Gate
コード変更後に「動くはず」「問題ないはず」は禁止。必ず証拠を示すこと:
1. `npx tsc --noEmit` の出力を貼る
2. `npx vitest run` の出力を貼る
3. 変更がテスト実行後に発生した場合は再テスト必須
「shouldは証拠ではない」— 検証なしのコミットは禁止。
```

このルールをCLAUDE.mdに書いてから、Claude Codeが「これで動くはずです」とだけ言って終わることがなくなりました。毎回テスト結果を見せてくれます。

## 4. 78問題の一括監査と並列修正

ある程度動くアプリができた段階で、Claude Codeに全コードベースの監査を依頼しました。

```
コードベース全体を監査して。UXの問題、セキュリティの穴、
アクセシビリティの不備、パフォーマンスのボトルネックを
すべてリストアップして、優先度をつけて。
```

結果: **78件の問題**が見つかりました。

これを優先度順にバッチ分けし、独立した問題を18エージェントで並列修正しました。

**並列エージェント運用で学んだこと:**
- 同じファイルを触るタスクは絶対に分けない
- 各エージェントにブランチを切らせる（後でマージ）
- マージ後は必ず全テスト実行
- それでもコンフリクトは起きる。完全に独立したタスクだけ並列化する

## 5. i18n — 8言語対応の実装

Claude Codeの翻訳品質は実用レベルです。ただし、以下のルールをCLAUDE.mdに設定しました:

```
- User-facing text: Always use t('key', lang) / tf('key', lang, ...args) from src/i18n.ts
- New i18n keys: Add for ALL 8 languages (ja, en, es, fr, de, zh, ko, pt)
- Terminology: User-facing "Snapshot" (not "Handoff"). Internal code keeps handoff
```

このルールなしだと、Claude Codeは新しいテキストを英語のハードコードで追加しがちです。「全テキストはi18n経由」を明文化したことで、翻訳漏れがほぼなくなりました。

## 6. CLAUDE.mdの設計パターン

Loreの開発を通じて最も価値があったのは、CLAUDE.md（プロジェクトルートに置く設定ファイル）の設計です。

効果が大きかった項目:

| カテゴリ | 効果 |
|---------|------|
| Verification Gate | 「動くはず」禁止 → 毎回テスト結果付き |
| Code Conventions | ハードコード防止、i18n必須化 |
| Safety Rules | 3回同じ失敗でSTOP、5修正ごとにリスク評価 |
| Don't リスト | `eslint-disable`禁止、`as any`禁止 |
| Auto-Execute Rules | 朝の挨拶でdaily-routine自動実行 |

**CLAUDE.mdは「AIチームの行動規範」**です。人間のチームにコーディング規約があるように、AIにもルールを明文化することで、出力品質が安定します。

## 7. バイブコーディングを超えるために

「バイブコーディング」（AIに任せて雰囲気で進める開発）は速いが脆い。Loreの開発で意識したのは:

1. **レビュー密度を上げる** — 小さなコミット単位でClaude Codeの出力を確認する
2. **「なぜ？」を聞く** — 実装だけでなく理由を理解する
3. **テストで担保する** — 理解が不完全でも、テストが通れば一定の品質を保証できる
4. **ルールを明文化する** — CLAUDE.mdにコーディング規約を書く
5. **監査を定期的に行う** — 機能追加後にコードベース全体の監査をClaude Codeに依頼する

完全に理解していないコードを出荷する不安は正直あります。でも、780本のテストが通り、78件の監査を通過し、型チェックもlintもパスしている。それは「理解」とは別の、検証による品質保証です。
